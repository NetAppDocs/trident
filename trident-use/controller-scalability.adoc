= Automatic backend configuration for AWS FSx for NetApp ONTAP
:hardbreaks:
:icons: font
:imagesdir: ../media/

[.lead]
Trident supports automatic backend creation for AWS FSx for NetApp ONTAP.
By creating a Kubernetes StorageClass with the required parameters, administrators can provision and manage FSx for NetApp ONTAP backends without creating separate backend configuration files.
This enables Kubernetes-native lifecycle management and reduces operational overhead.

== Key concepts and definitions

=== Automatic backend configuration

Automatic backend configuration allows Trident to create and manage storage backends based on StorageClass definitions.
When a StorageClass includes supported FSx for NetApp ONTAP parameters, Trident generates the corresponding backend internally.

Backend lifecycle is tied to the StorageClass.

=== StorageClass annotations

Trident uses annotations on the StorageClass to report backend creation status.
These annotations indicate whether backend configuration succeeded or failed and provide diagnostic information.

== Supported parameters

To enable automatic backend creation, the StorageClass must include the following parameters.

[cols="1,1,1,3",options="header"]
|===
| Parameter | Required | Type | Description

| `fsxFilesystemID`
| Yes
| string
| AWS FSx for NetApp ONTAP filesystem ID

| `storageDriverName`
| Yes
| string
| Trident storage driver, such as `ontap-nas` or `ontap-san`

| `credentialsName`
| Yes
| string
| AWS Secrets Manager ARN containing ONTAP credentials
|===

All parameter values must be specified as strings.

== Create a StorageClass with automatic backend creation

The following example creates a StorageClass that triggers automatic backend creation for AWS FSx for NetApp ONTAP.

[source,yaml]
----
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ontap-fsx-demo
  annotations:
    description: "StorageClass for AWS FSx for NetApp ONTAP with automatic backend creation"
provisioner: csi.trident.netapp.io
parameters:
  fsxFilesystemID: "fs-0abc123"
  storageDriverName: "ontap-nas"
  credentialsName: "arn:aws:secretsmanager:us-west-2:123456789012:secret:trident-creds"
allowVolumeExpansion: true
reclaimPolicy: Delete
volumeBindingMode: Immediate
----

After you apply this StorageClass, Trident creates the backend automatically.

== Verify backend configuration status

Trident reports backend creation status through annotations on the StorageClass.

=== View StorageClass annotations

[source,bash]
----
kubectl get storageclass ontap-fsx-demo -o yaml
----

=== Status annotations

[cols="1,3",options="header"]
|===
| Annotation | Description

| `trident.netapp.io/configuratorStatus`
| Backend creation result: `Success` or `Failed`

| `trident.netapp.io/configuratorMessage`
| Detailed success or error message

| `trident.netapp.io/configuratorName`
| Internal Trident configurator identifier

| `trident.netapp.io/managed`
| Indicates that the backend is managed by Trident
|===

=== Example successful configuration

[source,yaml]
----
metadata:
  annotations:
    trident.netapp.io/configuratorStatus: Success
    trident.netapp.io/configuratorMessage: Completed Trident backend configuration
    trident.netapp.io/configuratorName: tconf-nas-sc1
    trident.netapp.io/managed: "true"
----

== Success and failure behavior

=== Successful backend creation

When backend creation succeeds:

* The StorageClass becomes operational
* PersistentVolumeClaims can be created immediately
* Trident provisions volumes using the automatically created backend

=== Failed backend creation

When backend creation fails:

* Trident sets `configuratorStatus` to `Failed`
* `configuratorMessage` describes the failure
* Trident does not create a usable backend

Correct the issue and reapply the StorageClass.

== Create PersistentVolumeClaims

You can use the StorageClass to provision volumes for application workloads.

[source,yaml]
----
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: fsx-pvc
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: ontap-fsx-demo
  resources:
    requests:
      storage: 100Gi
----

== Add additional FSx for NetApp ONTAP file systems

You can attach additional FSx for NetApp ONTAP file systems to an existing StorageClass to expand capacity.

=== Update the StorageClass

[source,bash]
----
kubectl edit storageclass ontap-fsx-demo
----

Add the following annotation.

[source,yaml]
----
metadata:
  annotations:
    trident.netapp.io/additionalFsxNFileSystemID: '["fs-03cc1a718cddd6e248"]'
----

=== Behavior

When you add additional file systems:

* Trident attaches the new filesystem to the existing backend
* Trident extends available storage pools
* Existing workloads continue without interruption

== Caveats and limitations

The following limitations apply to automatic backend configuration:

* Backend lifecycle is tied to the StorageClass
* Configuration errors appear only in StorageClass annotations
* Backend changes require editing or reapplying the StorageClass

== WARNING: Deleting a StorageClass

Deleting a StorageClass can delete the underlying Trident backend.

This can:

* Disrupt storage connectivity
* Break running applications
* Cause permanent data access loss

Before deleting a StorageClass:

* Identify dependent workloads
* Confirm backend deletion behavior
* Plan a controlled migration if required

== Recommendations

NetApp recommends the following:

* Use automatic backend configuration for simplified onboarding
* Verify StorageClass annotations before deploying production workloads
* Track StorageClass changes using version control
* Test StorageClass deletion behavior in non-production environments
