= Automatic volume expansion

:imagesdir: ../media/

[.lead]
Automatic volume expansion enables Persistent Volumes provisioned by Trident to grow automatically when used capacity reaches a defined threshold. This capability reduces operational overhead and helps prevent application disruption caused by capacity exhaustion.

Automatic volume expansion is implemented using Autogrow Policies. An Autogrow Policy defines:

* The utilization threshold that triggers expansion
* The amount by which the volume grows
* The maximum size the volume can reach

Volumes increase in size automatically when the defined utilization threshold is exceeded. Volumes are never automatically reduced.

== Requirements

Before configuring automatic volume expansion, ensure that the following requirements are met:

* Trident 26.02 or later
* Role-based access control permissions to create `TridentAutogrowPolicy` custom resources
* StorageClasses configured with `allowVolumeExpansion: true`
* Supported ONTAP protocols:
** Network File System
** Internet Small Computer Systems Interface
** Fibre Channel Protocol
** Non-Volatile Memory Express over Fabrics

== Limitations

* ONTAP Non-Volatile Memory Express raw block volumes earlier than ONTAP 9.16.1 do not support automatic expansion.
* For storage area network volumes, if `growthAmount` is less than or equal to 50 mebibytes, Trident automatically increases the value to 51 mebibytes before resizing, provided the resulting size does not exceed `maxSize`.
* In brownfield environments, automatic expansion might not function for certain existing volumes due to volume publication migration behavior.
* When a volume reaches `maxSize`, no further expansion occurs.
* Supported protocols for automatic volume expansion:
** Network File System
** Internet Small Computer Systems Interface
** Fibre Channel Protocol
** Non-Volatile Memory Express over Fabrics

== Create an Autogrow Policy

Autogrow Policies enable automatic volume expansion when volumes reach a defined capacity threshold.

Ensure you have:

* Trident 26.02 or later installed
* Role-based access control permissions to create `TridentAutogrowPolicy` resources
* Understanding of workload growth requirements

An Autogrow Policy defines how volumes expand automatically when they reach a defined capacity threshold.

You can create Autogrow Policies at any point in your workflow:

* Before StorageClasses and volumes are created
* After StorageClasses exist
* After volumes are provisioned

This flexibility allows you to introduce automatic expansion without recreating existing resources.

=== Autogrow Policy specification

Autogrow Policies are Kubernetes custom resources defined as follows:

[cols="1,3,1,1,2,1",options="header"]
|===
| Field | Description | Format | Required | Example | Default

| name
| Unique policy identifier
| String
| Yes
| production-db-policy
| None

| usedThreshold
| Capacity percentage that triggers expansion
| Percentage string
| Yes
| "80%"
| None

| growthAmount
| Amount to grow when threshold is reached
| Percentage or size
| No
| "10%" or "5Gi"
| "10%"

| maxSize
| Maximum volume size limit
| Kubernetes quantity
| No
| "500Gi"
| Unlimited
|===

=== Create an Autogrow Policy

. Create a YAML file defining your Autogrow Policy.
. Apply the policy to your cluster.
. Verify successful creation.

[source,yaml]
----
apiVersion: trident.netapp.io/v1
kind: TridentAutogrowPolicy
metadata:
  name: standard-autogrow
spec:
  usedThreshold: "80%"
  growthAmount: "10%"
  maxSize: "500Gi"
----

----
kubectl apply -f autogrow-policy.yaml
kubectl get tridentautogrowpolicy standard-autogrow
----

Expected output:

----
NAME                USED THRESHOLD   GROWTH AMOUNT   STATE
standard-autogrow   80%              10%             Success
----

== Configuration examples

=== Conservative policy for production databases

[source,yaml]
----
apiVersion: trident.netapp.io/v1
kind: TridentAutogrowPolicy
metadata:
  name: production-db-policy
spec:
  usedThreshold: "75%"
  growthAmount: "20%"
  maxSize: "5Ti"
----

=== Log storage with fixed growth

[source,yaml]
----
apiVersion: trident.netapp.io/v1

metadata:
  name: dev-policy
spec:
  usedThreshold: "90%"
  growthAmount: "10Gi"
  maxSize: "100Gi"

== Policy states

[cols="1,3,2",options="header"]
|===
| State | Description | Action Required

| Success
| Policy validated and ready to use
| None

| Failed
| Validation errors detected
| Review and correct specification

| Deleting
| Deletion in progress
| Wait until volumes are disassociated
|===

== Manage Autogrow Policies

After creating Autogrow Policies, you can view, update, and delete them.

=== View Autogrow Policies

----
kubectl get tridentautogrowpolicy
kubectl describe tridentautogrowpolicy production-db-policy
----

=== Using the Trident command-line utility

----
tridentctl get autogrowpolicy
tridentctl get autogrowpolicy production-db-policy -o yaml
----

=== Update an Autogrow Policy

IMPORTANT: Changes affect all volumes currently using the policy.

----
kubectl edit tridentautogrowpolicy production-db-policy
----

=== Update considerations

* Changes apply immediately to all associated volumes.
* Expansion uses updated parameters during the next growth event.
* Restarting volumes or pods is not required.
* Test changes in non-production environments before applying broadly.

=== Delete an Autogrow Policy

----
kubectl delete tridentautogrowpolicy production-db-policy
----

If volumes reference the policy, it remains in the `Deleting` state.

=== Deletion states

[cols="1,3",options="header"]
|===
| Scenario | Behavior

| No volumes reference policy
| Deleted immediately

| Volumes reference policy
| Enters Deleting state

| All volumes disassociated
| Policy deleted
|===

== Storage Class configuration

You can associate an Autogrow Policy with a StorageClass using the `trident.netapp.io/autogrowPolicy` annotation.

=== Prerequisites

* StorageClass must include `allowVolumeExpansion: true`

=== Example: Network File System storage class

[source,yaml]
----
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ontap-nas-autogrow
  annotations:
    trident.netapp.io/autogrowPolicy: "standard-autogrow"
provisioner: csi.trident.netapp.io
allowVolumeExpansion: true
----

=== Example: Interface storage class

[source,yaml]
----
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ontap-san-autogrow
  annotations:
    trident.netapp.io/autogrowPolicy: "production-db-policy"
provisioner: csi.trident.netapp.io
allowVolumeExpansion: true
----

=== Remove policy from StorageClass

----
kubectl annotate storageclass ontap-gold trident.netapp.io/autogrowPolicy-
----

== Volume operations

=== Provision volume with StorageClass default policy

Create a PersistentVolumeClaim using a StorageClass that has an Autogrow Policy applied.

=== Create volume with custom Autogrow Policy

Add the annotation to the PersistentVolumeClaim:

[source,yaml]
----
metadata:
  annotations:
    trident.netapp.io/autogrowPolicy: "production-db-policy"
----

=== Disable Autogrow on a volume

----
kubectl annotate pvc database-pvc trident.netapp.io/autogrowPolicy="none" --overwrite
----

== Monitor Autogrow Policy usage

=== Check volumes using a policy

----
tridentctl get autogrowpolicy production-db-policy -o json
----

=== Check which policy a volume uses

----
kubectl get pvc database-pvc -o jsonpath='{.metadata.annotations.trident\.netapp\.io/autogrowPolicy}'
----

=== Monitor policy events

----
kubectl get events --field-selector involvedObject.kind=TridentAutogrowPolicy
----

== Troubleshooting

=== Policy shows Failed state

Correct specification errors and reapply.

=== Cannot delete policy

Ensure all volumes are disassociated before deletion.

=== Volume not expanding

* Confirm StorageClass has `allowVolumeExpansion: true`
* Verify `maxSize` not reached
* Check policy state

=== Frequent expansions

Increase `usedThreshold`.

=== TAGRI rejected

Adjust `maxSize` or `growthAmount`.

== Getting help

If you need assistance troubleshooting Autogrow Policies or automatic volume expansion, gather the following information before contacting support:

* Trident controller and node logs
* The Autogrow Policy definition
* PersistentVolumeClaim annotations
* Relevant Kubernetes events

To verify the installed Trident version:

----
tridentctl version
----

Providing this information helps support teams diagnose configuration, policy, and backend issues more efficiently.


== Frequently asked questions

=== Can I create an Autogrow Policy after volumes are provisioned?

Yes. Apply the policy to existing PersistentVolumeClaims using annotations.

=== Can I change the policy for existing volumes?

Yes. Update the annotation to reference a different policy.

=== Can multiple StorageClasses use the same policy?

Yes.

=== How do I disable Autogrow for a volume?

Set the annotation value to `none`.

=== How do I monitor Autogrow activity?

Use Kubernetes events and the Trident command-line utility.
